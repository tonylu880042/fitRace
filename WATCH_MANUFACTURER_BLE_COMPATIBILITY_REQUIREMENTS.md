# FitRaceStudio 賽事用穿戴裝置 BLE 資料需求與相容性檢核表

文件版本：1.0  
更新日期：2026-06-22  
適用對象：穿戴裝置／運動錶製造商、韌體團隊、整合測試團隊

## 1. 目的

本文件定義 FitRaceStudio 進行即時競賽時，穿戴裝置必須透過 Bluetooth Low Energy（BLE）提供的資料與行為。錶廠可依此文件比對既有通訊協定，確認能否直接接入，或列出需調整的韌體項目。

FitRaceStudio 以每位選手一條獨立的 BLE 資料流進行計分；資料在每次通知後即更新現場排行榜。競賽支援距離、時間、熱量及最大功率四種模式。

## 2. 相容性結論判定

| 結論 | 判定條件 |
| --- | --- |
| 可直接整合 | 裝置提供標準 FTMS Service（`0x1826`），可通知對應資料特徵，且能提供本文件「必要資料」欄位。 |
| 可經小幅調整整合 | 裝置資料語意正確但使用自訂 Service／Characteristic，或缺少熱量欄位；需提供 GATT 規格與封包範例，並由 FitRaceStudio 增加轉換器。 |
| 不適用於完整賽事 | 無法穩定提供累計距離或累計時間，或僅提供畫面／雲端資料而不能以 BLE 即時通知讀取。 |

「可被手機 App 讀取」不等於可整合。必須能由現場的 BLE Central 直接訂閱，且不依賴網際網路、專用 App 登入或使用者操作。

## 3. 必要 BLE 連線能力

### 3.1 優先方案：標準 FTMS

裝置應廣播並提供 **Fitness Machine Service**：

| 項目 | 要求 |
| --- | --- |
| Service UUID | `00001826-0000-1000-8000-00805f9b34fb`（短 UUID：`0x1826`） |
| 傳輸角色 | 穿戴裝置為 BLE Peripheral／GATT Server；FitRace Edge Node 為 BLE Central／Client。 |
| 資料傳送方式 | 以 Characteristic Notification 推送；不可要求 Edge Node 固定輪詢才取得即時數據。 |
| 訂閱方式 | 資料 Characteristic 必須有 Notify property，並支援以 CCCD（`0x2902`）開啟通知。 |
| 多裝置情境 | 每支錶必須可被獨立辨識與同時連線；同場不同選手的資料不可混在同一個通知流。 |
| 連線限制 | 不應要求 PIN、使用者確認、App 前景執行或雲端登入。若必須配對／加密，請先提供完整流程。 |

### 3.2 FitRaceStudio 目前直接訂閱的資料特徵

| 裝置類型 | FTMS Characteristic | UUID | 目前狀態 |
| --- | --- | --- | --- |
| 跑步機／跑步資料 | Treadmill Data | `0x2ACD` | 直接支援 |
| 室內自行車／風阻車 | Indoor Bike Data | `0x2AD2` | 直接支援 |
| 划船機／划船資料 | Rower Data | `0x2AD1` | 直接支援 |
| 跑錶、滑雪機、橢圓機或其他 | 自訂或其他 FTMS characteristic | 依設備而定 | 需先提供協定與封包確認；目前不可假設直接支援 |

若錶廠使用自訂 UUID，仍可整合；請提供第 8 節的資料。自訂格式的欄位定義、位元組序、縮放係數及 flags 必須完整提供。

## 4. 賽事資料需求

### 4.1 必要資料欄位

所有數值均代表**同一支錶／同一位選手**。距離、時間、熱量必須是累計值，不是本次封包的增量。

| 資料 | 單位／型別 | 用途 | 必要競賽 | 要求 |
| --- | --- | --- | --- | --- |
| 累計距離 | 公尺（m），數值 | 進度與名次 | 距離賽；時間賽的顯示／排名 | 從本次活動或賽前歸零後持續累加；不得在比賽中倒退或重置。 |
| 累計活動時間 | 秒或毫秒，整數 | 完賽時間、時間賽進度 | **所有賽制** | 必須持續遞增；若以秒提供，FitRaceStudio 轉為毫秒。解析度至少 1 秒。 |
| 即時功率 | 瓦（W），整數 | 最大功率賽、即時看板 | 最大功率賽 | 必須表示當下功率，不可為平均功率；靜止時應回報 `0`。 |
| 累計熱量 | kcal，數值 | 熱量賽 | 熱量賽 | 必須為裝置計算的累計熱量；不得以單包增量或每日總熱量取代。 |

### 4.2 建議資料欄位

| 資料 | 單位 | 用途／說明 |
| --- | --- | --- |
| 即時速度 | km/h | 即時看板顯示。FTMS 對應值通常為 `0.01 km/h`。 |
| 步頻／踏頻／划頻 | rpm 或 steps/min | 即時看板與營運診斷。請在協定中明確說明其定義。 |
| 心率 | bpm | 看板顯示；非賽事計分必要條件。 |
| 累計步數／划槳數 | count | 診斷與日後新增玩法；非目前計分必要條件。 |
| 裝置時間戳 | Unix epoch ms 或 uptime ms | 用於除錯與遺失通知分析；非目前計分必要條件。 |

### 4.3 各賽制的最低可行資料

| 賽制 | 必要資料 | 計分／完賽方式 |
| --- | --- | --- |
| 距離賽 | 累計距離、累計活動時間 | 距離達目標時，以首次達標的累計活動時間判定完賽時間。 |
| 時間賽 | 累計活動時間；建議累計距離 | 活動時間達設定秒數時結束；距離可作為看板與排名資料。 |
| 熱量賽 | 累計熱量、累計活動時間 | 熱量達目標時，以累計活動時間判定。 |
| 最大功率賽 | 即時功率、累計活動時間 | 在設定時間內取最高一次即時功率。 |

> 重要：目前系統雖可在未收到裝置熱量時，依功率與時間估算分數，但該公式不是生理學等級的 kcal。若要舉辦公平的熱量賽，錶廠必須提供累計熱量，並提供其計算定義。

## 5. FTMS 封包對照要求

以下欄位為 FitRaceStudio 目前 FTMS 解析器會讀取的欄位。未列欄位可以存在，但不得破壞 FTMS flags 所指定的欄位順序。

| Characteristic | FitRaceStudio 讀取欄位 | FTMS 資料單位／處理 |
| --- | --- | --- |
| Indoor Bike Data (`0x2AD2`) | instantaneous speed、instantaneous cadence、total distance、instantaneous power、heart rate、elapsed time | 速度除以 100 得 km/h；踏頻乘以 0.5 得 rpm；距離為 m；功率為 W；時間為 sec。 |
| Treadmill Data (`0x2ACD`) | instantaneous speed、total distance、heart rate、elapsed time | 速度除以 100 得 km/h；距離為 m；時間為 sec。 |
| Rower Data (`0x2AD1`) | stroke rate、stroke count、total distance、instantaneous power、heart rate、elapsed time | 划頻乘以 0.5 得 rpm；距離為 m；功率為 W；時間為 sec。 |

錶廠必須確認：

- 前 2 bytes 的 Flags 為 little-endian，並依 FTMS 規範正確指出哪些選填欄位存在。
- 選填欄位必須依標準規定的順序排列；即使 FitRaceStudio 不使用某欄位，解析器仍會依 flags 略過它。
- 使用 **Instantaneous Power**，不可將 Average Power 填入即時功率欄位。
- 使用 **Total Distance**，不可將單次移動距離填入累計距離欄位。
- 使用 **Elapsed Time** 時須說明重置條件與溢位處理。標準 16-bit 秒數約每 18.2 小時回繞；單場競賽前必須能歸零，或由裝置明確處理回繞。

## 6. 即時性、資料品質與活動生命週期

| 項目 | 必要要求 | 建議值／驗收方式 |
| --- | --- | --- |
| 通知頻率 | 運動中持續推送資料 | 建議 1–4 Hz；最慢不得低於每秒 1 次。 |
| 傳輸延遲 | 從量測到通知的延遲須穩定 | 建議 < 1 秒；95% 通知延遲 < 2 秒。 |
| 靜止狀態 | 停止運動後仍須能正確表達狀態 | 功率／速度應降為 0；累計值保持最後正確值且不倒退。 |
| 累計值單調性 | 距離、時間、熱量不可倒退 | 除非已明確進入新活動；比賽中不得重置。 |
| 遺失通知 | 短暫漏包不得造成資料跳回 | 下一包應含完整累計值，可自行修正前一包遺失。 |
| 斷線與重連 | 斷線後可重新訂閱 | 重連後應繼續同一活動的累計值，或提供可辨識的新活動／重置事件。 |
| 多錶干擾 | 同場多支裝置可被獨立連線 | 請提供已驗證的同時連線數、距離與 RF 環境條件。 |

## 7. 現行整合限制與需先確認項目

1. FitRaceStudio 現行 FTMS client 已直接解析距離、時間、即時功率、速度、頻率與心率；**FTMS 的 Expended Energy 欄位目前尚未轉成上游 `calories` 資料**。若賽事要使用裝置原生熱量，需在整合前補上此轉換並以真機驗證。
2. 對於非 `0x2ACD`、`0x2AD2`、`0x2AD1` 的資料 characteristic，不應以「FTMS」名義直接判定相容；必須先提供實際 GATT 表與封包。
3. 現行 BLE 資料來源以單一 characteristic 通知為前提。若距離、功率、時間分散在多個 characteristic，需確認同步方式與欄位的新鮮度。
4. 裝置的 BLE 名稱或位址必須能在場館端穩定辨識。若使用隨機 MAC 位址，請提供可靠的裝置識別方式（例如序號 characteristic）。

## 8. 請錶廠提供的回覆資料

請針對每一個要驗證的錶款／韌體版本，提供下列資料：

1. 型號、韌體版本、硬體版本與 BLE stack 版本。
2. Advertising 封包範例，以及是否廣播 `0x1826`。
3. 完整 GATT table：Service UUID、Characteristic UUID、properties、CCCD 支援與存取權限。
4. 運動中的原始通知封包至少 30 秒，包含：開始、穩定運動、暫停／靜止、恢復、結束各階段。
5. 每個 byte／bit 的定義：endianness、signedness、縮放係數、flags、單位與有效範圍。
6. 距離、時間、熱量的起算與重置規則；是否會因暫停、斷線、換頁、午夜或活動自動結束而歸零。
7. 同時連線能力、是否支援未配對連線、配對／加密要求，以及重新連線流程。
8. 已知限制：通知頻率、背景模式、手機／App 依賴、資料更新延遲、低電量行為。

## 9. 場測驗收清單

| 測試 | 通過標準 | 結果（錶廠填寫） |
| --- | --- | --- |
| 掃描 | Edge Node 可掃描到裝置，且可穩定識別正確錶款。 | ☐ 通過 ☐ 未通過 |
| 訂閱 | 可對指定 characteristic 開啟 notification，無需人工確認或 App。 | ☐ 通過 ☐ 未通過 |
| 距離 | 連續運動 5 分鐘，距離遞增、單位為 m，無倒退或賽中重置。 | ☐ 通過 ☐ 未通過 |
| 時間 | 連續運動 5 分鐘，時間遞增、與實際經過時間誤差可接受，無倒退。 | ☐ 通過 ☐ 未通過 |
| 功率 | 由低到高再停止，通知中的即時功率隨動且停止後回到 0。 | ☐ 通過 ☐ 未通過 |
| 熱量 | 連續運動時熱量累增；暫停與重連後語意符合第 6 節。 | ☐ 通過 ☐ 未通過 |
| 即時性 | 運動中連續 5 分鐘，每秒至少 1 筆通知，95% 延遲小於 2 秒。 | ☐ 通過 ☐ 未通過 |
| 斷線重連 | 主動斷線後重新連線與訂閱成功，累計資料的處理符合規格。 | ☐ 通過 ☐ 未通過 |
| 多裝置 | 在預期場館數量下同時連線，選手間無資料串流或錯配。 | ☐ 通過 ☐ 未通過 |

## 10. 交付前確認

錶廠請回覆下列結論並附上第 8 節資料：

- [ ] 本裝置提供標準 FTMS `0x1826`。
- [ ] 本裝置可透過 Notification 提供所需資料。
- [ ] 距離與時間為單一活動的累計值，且比賽中不會重置。
- [ ] 本裝置提供即時功率（若要參與最大功率賽）。
- [ ] 本裝置提供累計熱量及其計算定義（若要參與熱量賽）。
- [ ] 已完成第 9 節場測，並提供原始封包與測試結果。
